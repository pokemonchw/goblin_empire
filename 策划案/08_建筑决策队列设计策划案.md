# 建筑决策队列设计策划案

> 本文件是建设指挥板中“决策队列”的专项产品与实现规格。建筑数值、解锁条件以 `03_资源建筑职业系统.md` 和运行时定义为准；界面结构以 `05_界面交互与开发落地规格.md`、`07_完整交互设计策划案.md` 为准。发生冲突时，本文件只覆盖“哪些建筑进入队列、以何种顺序出现、如何解释推荐”这一职责，不改变建筑结算、购买资格、图鉴顺序或固定检查器操作。

## 1. 设计结论

建筑决策队列不是“把能看到的建筑按价格排序”，而是一个只读的建设参谋。它要从当前游戏状态中识别最迫切的经营问题，找出能够直接解决问题或推进下一阶段的建筑，并用玩家可验证的理由给出有限选择。

队列继续保留三个区段和最多七项的界面骨架，但重定义其语义：

| 区段 | 数量 | 回答的问题 | 入选条件 |
| --- | ---: | --- | --- |
| 当前目标 | 1 | “现在最值得围绕哪栋建筑组织生产？” | 正式解锁、尚不可立即建造、可通过当前生产链到达，且战略价值最高 |
| 可立即建造 | 3 | “手里的资源现在花在哪里最有效？” | 正式解锁、当前可支付、建造后不会触发未确认的致命风险 |
| 先处理 | 3 | “是什么阻止帝国继续建设？” | 建筑本身受阻，或建造后会造成劳力、口粮、转换原料等系统性风险；必须同时给出处理方向 |

界面可沿用现有“下一步”“需处理”标题以减少改造，但产品语义和空态文案应分别按“当前目标”和“先处理”验收。队列不显示内部得分、不自动购买、不自动摧毁、不替玩家调整职业、制作或启停建筑。

## 2. 当前问题诊断

当前实现位于 `game/ui/render.js` 的 `renderBuildingDecisionQueue`。它具有确定性和不重复的优点，但智能程度不足：

1. “下一步”只在 `unaffordable` 中按里程碑、等待秒数和 `designOrder` 排序。短等待的低价值扩建可能长期压过真正的阶段入口，也可能推荐正在被其他系统持续消耗价格资源的建筑。
2. “可立即建造”只优先里程碑。它不能区分住房告急、口粮下降、容量爆仓、劳力过载、生产链断料和普通效率提升，也不计算重复建筑的边际收益。
3. “需处理”只按容量、来源、劳力、前置四档排序。建筑为什么值得处理、哪个上游建筑能够处理、处理后是否仍有第二重阻断，都没有进入决策。
4. `preview` 与真正可处理的阻断混排。匿名预览通常不是当前行动，可能挤掉劳力过载或关键容量不足。
5. 同一路线的相近建筑可占满三个槽位，选择表面很多，实际是在重复表达同一个方案。
6. 推荐会随资源跨过支付线、每秒产量轻微波动而跳动。玩家来不及读完，固定检查器选择也容易失去上下文。
7. 系统只看“买得起”，不看“买完会怎样”。例如生产建筑可能让劳力总量过载，转换建筑可能缺少稳定输入，口粮相关负担可能扩大净亏损。
8. 没有把“解锁新系统的一座”和“继续堆叠第十座同类建筑”区分开，也没有识别容量建筑为目标建筑铺路的间接价值。

## 3. 设计目标与非目标

### 3.1 设计目标

- **有用**：优先解决会让生产停摆、人口受损或资源长期浪费的问题。
- **推进**：没有危机时，稳定指向阶段入口、生产链突破和新系统解锁。
- **可解释**：每个入选项都必须能用“现状 → 建筑效果 → 预期变化”说明，不能只说“推荐建造”。
- **确定性**：同一游戏状态必须得到同一队列；不得引入随机数或依赖 DOM 顺序。
- **稳定**：小幅 tick 波动不应让队列频繁换位；重大风险和新里程碑仍应立即插队。
- **保留选择**：最多三个可建方案应尽量覆盖不同经营意图，而不是给出三个近似答案。
- **诚实**：无法预测或未接入结算的数据必须明确降级，不得伪造“收益最高”。

### 3.2 非目标

- 不建立自动建造 AI，不替玩家花费资源。
- 不求解全局最优建筑序列，不进行跨数小时的穷举模拟。
- 不修改建筑价格、效果、解锁或摧毁规则。
- 不让内部排序成为新的隐藏难度；图鉴仍按 `designOrder` 排列。
- 不承诺精确回本时间。多资源、容量截断、天气和手动行动会使其产生误导。

## 4. 决策输入与纯派生结果

队列只读取权威状态和静态定义。任何推荐字段都是运行时纯派生结果，不进入存档。

### 4.1 必需输入

- `GameState`：资源库存与容量、瞬时净流量、建筑拥有/启用数、人口、住房、劳力、暂停状态、科技、职业和已解锁系统。
- `BuildingDefinition`：稳定 ID、价格曲线、效果、解锁、`routeId`、`effectTags`、`isMilestone`、`designOrder`。
- `BuildingViewModel`：可见状态、当前价格、缺口、等待时间、容量/来源阻断和劳力风险。
- 统一资源流分析：必须与资源浮窗读取同一个瞬时流量来源；不得为队列另写一套产量结算。
- 阶段摘要：仍由关键建筑确定性推导，但阶段只提供目标偏好，不写入进度。

### 4.2 新增派生契约

实现时应增加下列命名契约，具体可使用 JSDoc：

```text
EmpirePressureSnapshot
    foodPressure            口粮压力：safe / warning / critical
    housingPressure         住房压力：safe / warning / critical
    laborPressure           劳力压力：safe / warning / critical
    storagePressureById     各资源容量压力
    deficitResourceIds      当前持续净亏损资源
    stalledChainIds         因输入不足或容量截断而停滞的生产链
    activeStageId           当前建设阶段

BuildingDecisionProfile
    buildingId              建筑稳定 ID
    queueStatus             available / reachable / blocked / preview
    intentIds               该建筑满足的经营意图数组
    primaryIntentId         用于队列去重和文案的首要意图
    urgencyTier             0-4 整数紧迫度
    progressionTier         0-4 整数推进价值
    directnessTier          0-3 整数解题直接度
    efficiencyTier          0-3 整数边际效率档位
    riskTier                0-4 整数建造后风险
    waitBand                now / short / medium / long / unreachable
    bottleneck              唯一主要瓶颈及处理方向
    reasonTokens            受控理由数据，不存最终拼接文案

BuildingQueueSnapshot
    target                  0 或 1 个决策项
    available               0-3 个决策项
    attention               0-3 个决策项
    signature               用于防抖比较的稳定签名
```

`intentIds` 使用封闭集合：`survive_food`、`expand_housing`、`recover_labor`、`raise_capacity`、`open_source`、`stabilize_chain`、`increase_output`、`unlock_system`、`advance_stage`、`military_readiness`、`faith_growth`、`abyss_growth`。不能从中文描述猜测意图；优先由结构化效果映射，特殊建筑允许在静态表中显式声明受控提示字段。

## 5. 帝国压力诊断

先诊断帝国，再评价建筑。压力级别使用阈值和滞回，避免在边界来回切换。

### 5.1 口粮压力

口粮以菌菇当前净流量和安全储备时间共同判断：

- `critical`：菌菇净流量为负，按当前消耗预计 60 秒内耗尽；或库存已经为 0 且仍在净消耗。
- `warning`：菌菇净流量为负，预计 60-300 秒耗尽；或库存低于容量的 10%。
- `safe`：其他情况。

从 `critical` 恢复到 `warning` 必须连续 10 秒不再满足严重条件；从 `warning` 恢复到 `safe` 必须连续 15 秒净流量非负。暂停时保留最后一次有效压力，不推进持续时间。

### 5.2 住房压力

- `critical`：存活人口达到住房上限，或已经拥挤。
- `warning`：剩余住房不超过 1，且人口系统仍具备增长条件。
- `safe`：其他情况。

住房建筑只在实际增加 `housingMax` 时作为直接解决方案；兼带住房但成本极高的军事建筑仍可入选，但直接度低于纯住房方案。

### 5.3 劳力压力

- `critical`：减免后建筑占用超过人口劳力，生产建筑已经停摆。
- `warning`：剩余劳力小于下一个候选生产建筑的单座占用，或占用率达到 85%。
- `safe`：其他情况。

劳力过载时，降低劳力占用的建筑优先于新增生产建筑；新增零劳力建筑可以正常推荐。若减免建筑自身无法在当前过载状态下产生效果，文案必须说明其效果结算方式，不能假定立即恢复。

### 5.4 容量与供应链压力

- 资源库存达到容量 95% 且净流量为正，视为正在浪费产能；对应容量建筑获得紧迫度。
- 目标建筑价格超过某资源容量时，记录为容量硬阻断。能够提升该资源容量至目标价格以上的建筑是“直接解锁阻断”；只能部分抬高容量的是“铺路”。
- 资源净流量不为正不能一律视为“暂无来源”。手动采集、制作、贸易、掠夺、远征等离散来源应显示其真实来源类型；只有定义和已解锁系统都不存在取得路径时，才叫“暂无来源”。
- 转换建筑的输入资源若持续净流量不足以覆盖新增消耗，则标记为链路风险。风险进入“先处理”或降低可建优先级，而不是把建筑错误显示为不可购买。

## 6. 建筑价值评价

### 6.1 先硬门槛，后排序

候选评价顺序固定为：

1. 揭示和正式解锁检查；
2. 当前购买状态检查；
3. 建造后致命风险检查；
4. 当前压力匹配；
5. 阶段推进与系统解锁价值；
6. 边际收益和重复惩罚；
7. 等待档位与设计顺序兜底。

“建造后致命风险”包括直接造成劳力过载、使关键口粮从非负转为严重净亏损、或新增转换消耗没有可用上游。致命风险建筑不进入普通“可立即建造”，而进入“先处理”；检查器仍允许玩家知情建造。

### 6.2 使用分档序元，不使用伪精确总分

系统内部不把所有价值压成一个看似精确的浮点总分。每个建筑生成一个按顺序比较的决策键：

```text
[紧迫度降序,
 直接解决程度降序,
 阶段推进价值降序,
 首座/里程碑价值降序,
 边际效率档位降序,
 风险档位升序,
 等待档位升序,
 重复拥有惩罚升序,
 designOrder 升序]
```

这样既保证确定性，也能解释“为什么 A 在 B 前面”。`designOrder` 只能作为最终平局规则，不能代替策略。

### 6.3 紧迫度档位

| 档位 | 含义 | 示例 |
| ---: | --- | --- |
| 4 | 正在停摆或短期内造成不可逆损失 | 劳力过载导致全线停产；菌菇即将耗尽 |
| 3 | 已触顶或阻断明确目标 | 关键价格超过容量；人口已经满住房 |
| 2 | 近期效率损失或阶段链断点 | 资源持续爆仓；关键转换缺上游 |
| 1 | 一般成长机会 | 产量、服从、军力或容量的常规提升 |
| 0 | 当前没有可识别作用 | 仅保留图鉴，不进入有限队列 |

### 6.4 推进价值

- 4：首次建造后开放主标签、核心生产链或阶段，例如涂鸦墙、工匠棚、酋长厅、深渊门。
- 3：首次开放关键资源来源、职业、配方或系统入口。
- 2：解除当前目标的容量/来源硬阻断。
- 1：强化当前阶段主路线。
- 0：与当前阶段和压力无直接关系的常规叠加。

`isMilestone` 是推进价值的重要输入，但不能无条件压过生存危机。

### 6.5 边际价值与重复惩罚

- 首座建筑获得“能力从无到有”的优先档；后续建筑只评价下一座的真实增量。
- 容量类建筑按“新增容量中当前可利用部分”分档。库存和价格均远低于旧容量时，继续扩容属于低边际价值。
- 产出倍率类建筑应读取受影响资源的当前基础流量。基础流量为零时，倍率建筑不是直接来源，只能作为铺路项。
- 生产类建筑在输出资源已满且没有可用消耗端时降档；若它同时提供关键解锁，保留推进价值。
- 同一建筑拥有量越高，常规重复项惩罚越高；紧迫问题的直接解决方案和明确线性缺口不受机械惩罚。例如住房仍严重不足时，不能因为已建多座窝棚就停止推荐。

## 7. 三个区段的生成算法

### 7.1 公共候选池

1. 从统一 `BuildingViewModel` 收集全部已揭示建筑。
2. `hidden` 永不入队。
3. `preview` 默认不占七个正式槽位；只有队列所有正式区段均为空时，才在“先处理”显示一个匿名远景提示，并且不泄露数值。
4. 为正式建筑生成 `BuildingDecisionProfile`。
5. 同一建筑只能分配到一个区段，分配顺序为“先处理”致命风险 → “可立即建造” → “当前目标” → 其他阻断。

### 7.2 当前目标：最多 1 项

候选必须是 `reachable`：当前不可支付，但所有缺口都能通过当前已启用的持续来源或已开放的明确离散来源获得，且价格不超过容量。等待时间为无穷、来源未知或容量不足的建筑不进入当前目标。

选择时先比较紧迫度和推进价值，再比较等待档位。等待时间只在战略价值接近时决定先后，避免“便宜但无关”的建筑永远占据目标。

等待档位固定为：短期不超过 60 秒，中期 60-300 秒，长期超过 300 秒。长期目标仅在它是推进档位 3-4 或当前没有更近的有效目标时入选。

推荐理由示例：

- “当前目标：涂鸦墙。首座将开放研究；还缺 9 朽木，按当前流量约 00:45。”
- “当前目标：窝棚。人口已住满；还缺 3 朽木，可通过捡柴获得。”

### 7.3 可立即建造：最多 3 项

从可支付且无致命风险的候选中选择，采用“意图覆盖”而不是直接取排序前三：

1. 先取全局最高决策键候选。
2. 第二项优先选择不同 `primaryIntentId`；没有时选择不同 `routeId`。
3. 第三项继续优先未覆盖意图；确无替代时才允许同意图重复。
4. 同一建筑只能出现一次；不同建筑即使共享路线，只要解决不同压力仍可并列。

危机期间允许打破多样性：若口粮、住房或劳力同时为 `critical`，每个严重问题都可占一个槽位。没有危机时，理想的三项分别表达“阶段推进”“解除瓶颈”“效率成长”。

### 7.4 先处理：最多 3 项

该区段显示“值得关注且能说清下一步”的问题，而不是罗列所有买不起的建筑。优先级为：

1. 当前已经发生的全局停摆；
2. 可购买建筑的建造后致命风险；
3. 当前目标的容量硬阻断；
4. 当前目标的来源/生产链阻断；
5. 具名解锁前置；
6. 匿名预览兜底。

每项必须提供一个唯一主瓶颈和一个行动方向：

- 容量不足：“铁片容量 50，下一座需 75；先扩建可增加铁片容量的仓储。”
- 来源不足：“金币当前无持续来源；可通过已开放贸易获得，不计算自动等待时间。”
- 劳力风险：“建造后占用将达到 112/100，生产线会停摆；先增加人口或建劳力减免设施。”
- 链路风险：“深炉会新增焦油消耗，当前焦油净流量不足；先稳定焦油井产出。”
- 前置不足：“需先研究‘采矿’；建筑队列不替代研究页操作。”

如果系统无法从定义中找到真实处理方向，显示“当前未发现可用取得路径”，并在开发环境暴露配置缺口；不得退化为笼统的“暂无来源”。

## 8. 间接解题与依赖回溯

智能队列必须能回答“目标建筑造不了，应先造什么”。回溯深度限制为两层，避免产生难以解释的长链规划。

### 8.1 回溯规则

1. 选出最高价值但受容量或来源阻断的目标建筑。
2. 对每个主要瓶颈资源查找能够直接增加容量、直接生产、解锁生产来源或降低相关消耗的建筑。
3. 若解决建筑可建或可达，把它作为铺路候选，并在理由中引用原目标。
4. 若解决建筑自身又被阻断，只允许再回溯一层；之后停止并显示最靠近玩家的可行动节点。
5. 多个解决建筑并存时，按“能否一次解除 → 风险 → 等待档位 → 边际价值 → `designOrder`”排序。

示例：深渊门缺魔晶且容量也不足时，队列先推荐真正抬高魔晶容量的设施；若容量够但没有魔晶来源，再推荐解锁或产生魔晶的上游。不能直接把深渊门以“暂无来源”挂在首位数小时。

### 8.2 防循环

依赖回溯维护本次计算的建筑 ID 与资源 ID 访问集合。遇到重复 ID、互相依赖或超过两层时立即停止，生成 `dependency_cycle` 配置诊断；玩家文案只说“当前生产链尚未打通”，开发控制台给出稳定 ID 链路。

## 9. 队列稳定与刷新策略

### 9.1 两级刷新

- **模型刷新**：模拟 tick 后可以重新计算压力、价格和文案数值。
- **成员刷新**：只在结构事件或稳定窗口满足时替换队列成员。

结构事件包括建造/摧毁建筑、完成科技、解锁资源或系统、人口变化、劳力从正常跨入过载、资源从安全跨入即将耗尽、加载存档和迁徙开局。结构事件允许立即重排。

普通资源波动只更新缺口和等待文案。候选必须持续优于当前成员 10 秒，且至少高出一个主要档位，才替换成员。当前成员变为不可见、已经无任何作用或跨入致命风险时立即移出。

### 9.2 选择保护

- 玩家当前选中的建筑即使暂时离开七项队列，固定检查器也保持选择，直到建筑变为 `hidden` 或玩家另选。
- 悬停/聚焦浮窗期间继续保留列表 DOM；只更新安全的文本节点或延迟成员刷新。
- 建造成功后保持当前选择，并立即重算下一座边际价值；不得因为拥有量变化强制跳到另一建筑。

### 9.3 防抖状态不写存档

候选优胜持续时间、上次队列签名和界面选择属于运行时偏好。加载后可立即按当前权威状态生成新队列，不需要迁移存档版本。

## 10. 玩家可见信息

蓝图签仍保持高密度，只显示：状态字形、名称/拥有量、一个主效果、行动结论、唯一瓶颈。智能依据通过短句表达，不显示“紧迫度 4”“得分 87”或权重公式。

### 10.1 文案结构

统一使用以下结构之一：

- 危机：`住房已满 → 建成后增加 2 住房`
- 推进：`首座开放研究 → 当前可建`
- 铺路：`为工匠棚补足碎石容量 → 还缺 40 朽木`
- 风险：`建造后劳力 112/100 → 先增加人口或降低占用`
- 常规成长：`菌菇持续产出 +X → 当前容量可承接`

“唯一瓶颈”由决策层产出结构化原因，渲染层只负责查中文名和格式化数值。若存在多个缺口，选择规则为：致命风险 → 容量硬阻断 → 无来源 → 最大等待贡献 → `Price[]` 顺序；检查器仍显示完整缺口列表。

### 10.2 空态

- 当前目标为空：“没有需要等待的明确目标；可直接建设或前往图鉴查看。”
- 可立即建造为空：“当前没有安全且可支付的建设方案。”
- 先处理为空：“当前未发现建设阻断或建造后严重风险。”
- 七项全空且只有预览：“继续探索资源或完成具名前置，新的建设方向将逐步显现。”

## 11. 特殊场景规则

- **暂停**：保持队列成员和诊断，所有行动显示已暂停；等待时间不倒计时，也不因暂停把全部候选降为普通受阻。
- **新局**：优先口粮、住房、首个研究入口；不能因菌菇床便宜而让三个槽位全是重复生产建议。
- **劳力过载**：除零劳力建筑、住房和劳力减免方案外，生产建筑不进入普通可建区。
- **资源满仓**：若输出资源已满，新增该资源生产建筑降档；可扩容或有真实消耗端的建筑升档。
- **离散来源**：手动采集显示“可手动取得”，制作显示配方，贸易/掠夺/远征显示对应系统；不伪造每秒等待时间。
- **挑战模式**：读取挑战修正后的真实流量、容量和成本。禁用系统相关建筑不得被推荐为解法。
- **负面效果建筑**：新增消耗或风险必须进入风险档位；若其推进价值很高，可进入“先处理”等待知情决策。
- **一次性里程碑已建**：如果重复建造不再提供解锁价值，后续只按实际增量评价，不能永久享受里程碑优先级。
- **全建筑均不可达**：展示最关键的三个可行动阻断；若没有任何可行动路径，明确引导到研究、制作、外交等对应页面，不循环推荐建筑自身。

## 12. 实现落地建议

保持现有简单静态浏览器架构，不引入框架或异步规划器。建议按以下职责拆分：

1. `game/systems/buildingView.js` 继续负责单栋建筑的公开状态、价格和风险基础数据。
2. 新增 `game/systems/buildingDecisions.js`，只负责压力快照、意图映射、决策档位、依赖回溯和三个区段的纯派生结果。
3. `game/ui/render.js` 只消费 `BuildingQueueSnapshot`，不在渲染函数中编写排序业务规则。
4. `game/scripts/contracts.js` 增加上述派生契约；它们不进入 `SaveData`。
5. 若现有效果字段不足以可靠识别意图，在 `BuildingDefinition` 增加显式 `decisionHints` 受控字段，例如 `intentIds`、`stageIds`、`riskResourceIds`。该字段只补充无法从结构化效果推导的设计意图，不能复制效果数值。
6. 资源来源查询应抽为公共纯函数，供建筑队列、固定检查器和资源浮窗复用，消除“无持续来源”与“完全不可取得”的混淆。

第一版无需做经济模拟器。先覆盖口粮、住房、劳力、容量、来源、里程碑、首座价值和路线多样性；第二版再接入转换链输入覆盖与两层依赖回溯；第三版根据真实游玩日志微调阈值，不改变排序字段语义。

## 13. 验收矩阵

### 13.1 功能案例

| 场景 | 输入状态 | 必须观察到的结果 |
| --- | --- | --- |
| 新局可建多座菌菇床 | 基础资源充足，无住房余量 | 住房方案压过第三个同质产出建议；可建区不被单一路线占满 |
| 菌菇即将耗尽 | 净流量为负，预计 60 秒内归零 | 直接改善口粮的建筑进入最高优先；普通里程碑不能压过危机 |
| 人口住满 | 存活人口等于住房上限 | 可建住房进入可建区首位，或可达住房成为当前目标 |
| 劳力已经过载 | 建筑占用大于人口劳力 | 新生产建筑移入先处理；减免、住房/人口铺路或零劳力方案优先 |
| 里程碑短期可达 | 涂鸦墙未建，资源约 45 秒可齐 | 它成为当前目标，并说明开放研究和准确缺口 |
| 低价值建筑更快 | 普通扩建 10 秒可达，里程碑 45 秒可达 | 若无危机，里程碑仍可优先；等待时间不单独决定结果 |
| 价格超过容量 | 目标需 75 铁片，容量 50 | 目标进入先处理；真正增加铁片容量的建筑成为铺路候选 |
| 只有贸易来源 | 金币净流量为 0，但贸易已开放 | 文案说明可通过贸易取得，不显示“暂无来源”或虚假等待秒数 |
| 转换输入不足 | 深炉可支付，但焦油净流量不足 | 深炉不进入普通安全可建，先处理说明焦油链风险 |
| 输出已满 | 铁矿满仓且没有消耗端 | 新增铁矿产出建筑降档，扩容或消耗端优先 |
| 重复里程碑 | 已拥有涂鸦墙，再次可建 | 不再按“开放研究”推荐，只按下一座真实增量评价 |
| 多个可建方案 | 阶段、容量、效率建筑同时可建 | 三槽优先覆盖不同意图，不机械取同一路线前三项 |
| 匿名预览很多 | 正式候选存在，另有多个 preview | preview 不挤占正式七槽；浮窗继续遵守匿名规则 |
| 小幅流量波动 | 两候选在支付线附近反复变化 | 队列成员不逐 tick 交换；缺口文案可以更新 |
| 重大风险出现 | 人口死亡导致劳力过载 | 不等待防抖窗口，先处理区立即更新 |
| 暂停游戏 | 当前已有目标和等待时间 | 成员保持，等待不推进，检查器操作显示已暂停 |

### 13.2 不变量

- 同一权威状态重复计算 100 次，队列成员和顺序完全一致。
- 三个区段合计不超过 7 项，同一 `buildingId` 不重复。
- `hidden` 永不出现；`preview` 不泄露名称、成本、效果数值。
- 队列计算不修改 `GameState`、建筑拥有量、职业、资源或存档版本。
- 所有中文资源、建筑、科技、配方和标签名通过稳定 ID 查询定义表。
- 图鉴的 `designOrder`、筛选和不可选择语义不受智能排序影响。
- 建造资格仍由建筑系统在点击时最终复核；队列结果不能绕过支付、暂停或解锁检查。
- 所有内部决策键最终以 `designOrder` 和稳定 ID 兜底，不依赖对象枚举顺序。

### 13.3 开发验证

为纯决策模块建立固定状态夹具，至少覆盖验收矩阵前 12 项，并断言区段、首项、主意图和瓶颈类型。手工回归还需检查桌面固定检查器、窄屏底部抽屉、悬停防闪烁、键盘选择和建造后保持选择。

开发环境可选择性输出结构化诊断：候选 ID、决策键、淘汰原因和依赖链。正式界面不得显示分数，但调试信息必须足以解释排序错误。

## 14. 完成定义

本方案落地完成必须同时满足：

1. 排序业务已从渲染层移入纯决策模块；
2. 队列能够识别口粮、住房、劳力、容量、来源和生产链风险；
3. 当前目标不再由最短等待时间单独决定；
4. 可建区具备意图多样性和建造后风险过滤；
5. 先处理区能给出真实、唯一且可行动的瓶颈方向；
6. 具备结构事件即时刷新、普通波动防抖和选择保护；
7. 自动测试覆盖核心状态夹具，手工回归通过现有建设指挥板交互；
8. 不改变自动化边界、存档兼容、图鉴顺序和建筑结算权威。

满足这些条件后，建筑决策队列才能从“稳定的状态列表”升级为“理解当前帝国问题、能提出有限且可信建设方案的参谋”。

## 15. 第二版：战略目标、行动节点与资源保留

第一版压力分档解决了“眼前哪栋更急”，但逐栋评价仍无法回答两个高频问题：一是“这个仓库为何比另一座产出建筑更值得现在建”，二是“明明正在攒里程碑，为什么队列还推荐花掉同一批材料”。第二版在既有序元排序之前增加一个有限规划层，不做经济穷举，也不修改建造结算。

### 15.1 三层决策模型

每次模型刷新依次形成三层只读结果：

1. **帝国压力层**：口粮、住房、劳力和正在浪费的容量决定不可延后的危机；该层沿用第 5 节，拥有最高裁决权。
2. **战略目标层**：在尚不可立即支付、`progressionTier >= 2` 的正式建筑中，按“推进价值、紧迫度、直接度、等待档位、风险、设计顺序、稳定 ID”选出一个主目标。目标选择刻意先看推进价值，避免某个普通建筑仅因自身价格超过容量就冒充帝国主线；眼前危机已由更高权重的压力层处理。只保留一个主目标，避免同时为多个远景互相矛盾地预留资源。
3. **行动节点层**：从全部正式建筑查找能直接解决主目标容量或来源阻断的最近建筑；同时识别普通可建项对主目标缺口库存的挪用程度。

战略层只观察当前已揭示定义、当前价格和当前库存。未揭示内容不得参与规划，系统不能利用玩家尚不知道的信息形成“神谕式推荐”。主目标已可支付后退出资源保留阶段，由可立即建造区正常比较。

### 15.2 战略杠杆档位

`BuildingDecisionProfile` 新增 `strategicLeverageTier`，取值为 0-3：

| 档位 | 含义 | 例子 |
| ---: | --- | --- |
| 3 | 下一座即可一次解除主目标的容量硬阻断，或建立目标资源的首个直接来源 | 建一座铁片仓后容量从 50 达到门槛 75 |
| 2 | 能部分抬高所需容量，仍需继续建设；或属于可验证的近端铺路 | 一座仓库只能把容量从 50 提至 60 |
| 1 | 只提供间接效率，不能证明会解除当前阻断 | 提高已有矿场倍率但目标缺的是铁片来源 |
| 0 | 与主目标没有结构化关系 | 普通住房扩建与工业里程碑无关 |

档位 3-2 必须由效果字段与阻断资源 ID 精确相交产生，不能从中文描述或路线名称猜测。容量解法必须计算“现容量 + 下一座增量”是否达到目标当前价格；来源解法必须存在对应的正向直接产出字段。行动节点理由固定为“为‘目标名’解除资源名阻断”，使玩家能核对因果。

比较顺序调整为：紧迫度、直接度、战略杠杆、推进价值、资源冲突、首座/里程碑、边际效率、风险、等待、重复惩罚、`designOrder`、稳定 ID。这样危机仍压过长期计划，但真正的铺路建筑不会被一个泛化的“首座产出”标签压住。

### 15.3 目标资源保留

`BuildingDecisionProfile` 新增 `reservationConflictTier`，取值为 0-2。先按主目标当前价格减去库存，得到 `missingAmountByResourceId`；只对仍有缺口的资源建立保留，不冻结已经足额的材料。

- 0：候选价格不消耗任何目标缺口资源。
- 1：候选会消耗目标缺口资源，但单项消耗低于该资源当前库存的 50%。
- 2：候选单项消耗达到该资源当前库存的 50% 或以上，购买会明显推迟主目标。

资源保留是排序建议，不是支付锁。玩家仍可在检查器知情建造。紧迫度 3-4 的住房、生存和停摆解法不应用资源冲突降序；战略铺路节点也先按杠杆处理，避免“为了保留材料而拒绝建造解除容量的必要仓库”。玩家可见理由为“会占用‘目标名’所需资源 → 非紧急时暂缓”。

### 15.4 规划失效与重算

发生以下任一情况时立即废弃旧规划并重算：主目标完成或变为可支付、目标隐藏、建筑建造/摧毁、资源或系统正式解锁、人口变化、压力跨级。普通库存变化允许重算缺口和冲突档位，但成员仍遵守 10 秒稳定窗口。资源保留状态与主目标均为运行时派生信息，不写入存档。

若没有 `progressionTier >= 2` 的未支付正式建筑，则不建立主目标，也不施加资源保留惩罚。若目标存在但没有结构化阻断，只保留既有“当前目标”行为，不臆造铺路节点。

### 15.5 第二版验收补充

| 场景 | 必须观察到的结果 |
| --- | --- |
| 里程碑价格超过铁片容量，仓库与矿场均可建 | 能一次抬够容量的仓库排在矿场前，并具名说明为里程碑铺路 |
| 仓库一座只能部分补容量 | 仓库仍可作为档位 2 铺路，但不得声称一次解除 |
| 正在积累里程碑所缺朽木，两个普通建筑均可建 | 不消耗朽木者排在大量消耗朽木者前；后者说明资源冲突 |
| 菌菇 60 秒内耗尽，但口粮建筑消耗里程碑材料 | 口粮危机仍优先，资源保留不得压过生存 |
| 主目标已可支付 | 不再以该目标对其他建筑施加资源保留惩罚 |
| 未揭示建筑恰好能解决阻断 | 不得使用该建筑作为行动节点或在理由中泄露名称 |

自动测试必须至少覆盖“一次解除的战略杠杆”和“同资源显著挪用”两项，并继续满足第 13.2 节全部不变量。

## 16. 第三版：目标完成度、多阻断覆盖与时间机会成本

第二版能够识别一个战略目标，却仍可能把“碰巧命中一个字段”误当成最佳行动。第三版不扩大自动化权限，而是在既有有限规划层中补齐三个可验证维度：计划离落地还有多远、下一座能同时消除多少阻断、现在花掉材料会让目标实际延迟多久。

### 16.1 战略目标完成度

同一推进档位、紧迫度和直接度的目标，按当前价格的加权完成比例选择更接近落地者。对每项价格计算 `min(库存, 需求)`，再以全部需求量归一化为 `goalCompletionRatio`，范围为 0-1。该比例只在相同战略等级内打破平局，不能让接近完成的普通扩建压过核心系统入口，也不能压过生存危机。

选择目标后，蓝图签仍只显示一个主目标，不显示比例分数。完成度属于运行时派生字段，不写入存档；价格、库存、解锁或建筑数量发生变化时重新计算。

### 16.2 多阻断覆盖

目标可能同时缺少多项容量或来源。每个铺路候选必须遍历目标的完整阻断集合，生成 `blockerCoverageCount`：下一座能够结构化处理的不同阻断资源数量。

- 只有当下一座覆盖目标的全部当前阻断，且每项容量均能达到价格门槛或来源均可建立时，`strategicLeverageTier` 才为 3。
- 只覆盖部分阻断、只能部分扩容，或覆盖全部但不能一次清除时为 2。
- 同杠杆档位内，覆盖两项的建筑优先于只覆盖一项的建筑；风险、可达性与边际价值继续参与后续比较。
- 玩家文案显示一个主要资源，并在同时覆盖多项时补充“等 N 项”；检查器继续展示完整成本与风险，避免蓝图签过载。

该规则禁止把“建立任意正产出”无条件描述为完全解决。来源建筑若自身会造成劳力过载或输入链断裂，仍进入“先处理”，不能因覆盖数高绕过风险门槛。

### 16.3 时间机会成本

资源保留从库存比例升级为恢复时间。对候选消耗的每项目标缺口资源，读取统一资源流分析的真实正向净流量，估算 `候选消耗量 / 正向净流量`，多资源取最长值作为 `reservationDelaySeconds`。

| 延迟 | 冲突档位 | 决策含义 |
| ---: | ---: | --- |
| 少于 10 秒 | 0 | 可快速补回，不为此压低正常成长方案 |
| 10-30 秒 | 1 | 有可感知延迟，同级候选中后置 |
| 30 秒及以上 | 2 | 明显打断计划，非紧急时显著后置 |
| 无正向自动流量 | 2 | 当前无法自动补回，必须明确提示 |

时间机会成本仍是建议而非支付锁。紧迫度 3-4 的生存、住房、劳力解法和战略铺路节点不受普通资源保留压制。离散来源不伪造自动秒数；其自动恢复时间视为无穷，并在文案中显示“当前无法自动补回”。

### 16.4 第三版确定性决策键

普通候选比较键更新为：

```text
[紧迫度降序,
 直接解决程度降序,
 战略杠杆降序,
 阻断覆盖数量降序,
 推进价值降序,
 资源冲突档位升序,
 资源恢复延迟升序,
 首座/里程碑价值降序,
 边际效率降序,
 风险升序,
 等待档位升序,
 重复惩罚升序,
 designOrder 升序,
 稳定 ID 升序]
```

战略目标比较键在推进价值、紧迫度、直接度之后加入完成度降序。所有字段均来自权威状态或结构化定义；不得读取中文描述、DOM 顺序或随机数。

### 16.5 第三版验收补充

| 场景 | 必须观察到的结果 |
| --- | --- |
| 两个同级里程碑分别完成 80% 与 20% | 80% 目标优先成为唯一战略目标；不得跨越更高推进档位 |
| 目标同时受碎石与铁片容量阻断 | 同时扩两项容量的设施优先于只扩一项的设施 |
| 综合仓库只能部分抬高两项容量 | 覆盖数为 2，但杠杆不得宣称“一次全部解除” |
| 消费仅占库存 20%，但需 100 秒补回 | 判为高资源冲突，不能因“不到库存一半”放行 |
| 消费占库存 60%，但 5 秒可补回 | 不因库存比例机械判为高冲突 |
| 铺路建筑覆盖两项但会劳力过载 | 进入先处理并说明劳力风险，不进入安全可建区 |
| 资源没有正向自动流量但可贸易获得 | 不显示虚假等待秒数；说明无法自动补回并保留贸易方向 |

自动测试至少覆盖多阻断覆盖与恢复时间机会成本，并继续满足第 13.2 节不变量。
